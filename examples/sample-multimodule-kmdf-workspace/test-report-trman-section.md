# Trace Manifest Embedding via PE Section — Test Report

**Date:** March 18, 2026  
**Project:** windows-drivers-rs  
**Feature:** Link-section based trace manifest embedding (`.trman` PE section)

---

## Architecture Summary

### Previous Approach
- Proc-macro atexit handlers write intermediate `.tmf` files to a scratch directory
- Driver crate's atexit handler reads dependent crate TMF files + its own entries
- PDB is patched with S_ANNOTATION symbols from the merged entries

### New Approach (Additive)
The previous TMF-based approach is **retained** for backward compatibility and PDB patching. In addition:

- Each `trace!()` macro expansion generates a const static `TraceManifestRecord` placed into the `.trman$m` PE section via `#[link_section = ".trman$m"]` + `#[used]`
- The `#[driver_entry]` macro generates start/end sentinels in `.trman$a` and `.trman$z`
- The MSVC linker merges all `$`-suffixed subsections alphabetically into a single contiguous `.trman` section
- The `.trman` section serves as the canonical in-binary trace manifest, enabling post-link extraction and verification

### Key Files Changed
| File | Change |
|------|--------|
| `crates/wdf/src/api/core/trace_manifest.rs` | **NEW** — `TraceManifestRecord`, `TraceArgRecord`, `fixed_str()`, sentinel constants |
| `crates/wdf/src/api/core/mod.rs` | Added `pub mod trace_manifest;` |
| `crates/wdf/src/lib.rs` | Added `__internal` re-exports for trace manifest types |
| `crates/wdf-macros/src/lib.rs` | `trace!()` generates `#[link_section = ".trman$m"]` statics; `#[driver_entry]` generates `.trman$a`/`$z` sentinels |
| `crates/wdf-macros/src/manifest.rs` | Stale TMF cleanup when crate has no traces |

## Test Results

### Test Environment
- **Target:** sample-multimodule-kmdf-workspace (4 crates: driver + 3 libraries)
- **Baseline:** 33 traces (22 driver crate + 4 static-lib + 3 dynamic-lib + 4 external-lib)
- **PDB tool:** llvm-pdbutil
- **Section tool:** llvm-readobj

### Scenario Matrix

| # | Scenario | PDB Annotations | .trman Section | Binary Size | Status |
|---|----------|----------------|----------------|-------------|--------|
| 1 | **Clean build** | 33 S_ANNOTATION (22+11) | 0x10798 (67,480 B) | 35 records × ~1928 B | ✅ PASS |
| 2 | **Incremental (no changes)** | 33 S_ANNOTATION | unchanged | unchanged | ✅ PASS |
| 3 | **Add trace to dependent crate** (sample-static-lib +1) | 34 S_ANNOTATION (22+12) | grew | +1 record | ✅ PASS |
| 4 | **Remove trace from dependent crate** (revert scenario 3) | 33 S_ANNOTATION (22+11) | shrunk back | -1 record | ✅ PASS |
| 5 | **Update trace format string** (sample-dynamic-lib) | 33 S_ANNOTATION (7052 B vs 7040 B) | updated | format text changed in PDB | ✅ PASS |
| 6 | **Remove ALL traces from crate** (sample-dynamic-lib → 0 traces) | 33 S_ANNOTATION* | 0xF100 (61,696 B) ← shrunk by 3 records | Correct: 3 records removed | ⚠️ PARTIAL |
| 7 | **Add trace to driver crate** | 34 S_ANNOTATION (23+11) | grew | +1 record | ✅ PASS |
| 8 | **Remove trace from driver crate** | 32 S_ANNOTATION (21+11) | shrunk | -1 record | ✅ PASS |
| 9 | **Single-crate driver** (sample-kmdf-safe-driver) | 14 S_ANNOTATION | 0x7880 (30,848 B) | 16 records (14+2 sentinels) | ✅ PASS |

### Scenario 6 Detail (⚠️ PARTIAL)
When **all traces** are removed from a dependent crate:
- **`.trman` section:** ✅ **CORRECT** — the 3 records from sample-dynamic-lib are absent from the binary (section shrunk from 67,480 → 61,696 bytes, exactly 3 × 1,928 = 5,784 bytes less)
- **PDB annotations:** ⚠️ **Stale TMF** — the old `sample_dynamic_lib.tmf` file persists because when NO `trace!()` calls exist, the proc-macro is never invoked and the atexit handler is never registered. The driver crate reads the stale TMF and includes those entries in the PDB.
- **Root cause:** The atexit handler is registered by `ensure_flusher_registered()` which is called only from `trace!()` macro expansion. When a crate removes all `trace!()` calls, no atexit runs.
- **Mitigation added:** `flush_manifest()` now cleans up stale TMF files when it runs with empty entries (handles the case where `trace!()` calls still exist elsewhere in the same compilation unit). Full fix would require registering the atexit from the `use wdf::trace;` import or build.rs.
- **Impact:** The `.trman` section (the new binary-embedded approach) is **always correct** since `#[link_section]` statics are only generated by actual `trace!()` calls.

## `.trman` Section Layout

```
.trman$a  →  [START sentinel: magic=0xAAAAAAAA]     ← from #[driver_entry]
.trman$m  →  [record 1] [record 2] ... [record N]  ← from trace!() calls
.trman$z  →  [END sentinel:   magic=0x5A5A5A5A]    ← from #[driver_entry]
```

### TraceManifestRecord Layout (1,928 bytes)
```
Offset   Size    Field
0        4       magic (0x5452_4D4E = "TRMN")
4        4       version (1)
8        4       message_id
12       4       line_number
16       4       arg_count
20       4       _reserved
24       64      crate_name (NUL-padded)
88       256     source_file (NUL-padded)
344      512     format_string (NUL-padded)
856      32      flag_name (NUL-padded)
888      16      level_name (NUL-padded)
904      16×64   args[16] (each: 32 name + 24 item_type + 4 param + 4 pad)
```

## Advantages of `.trman` Section Over TMF Files

| Aspect | TMF Files | `.trman` Section |
|--------|-----------|-----------------|
| Cross-crate aggregation | Manual: collect TMF files from scratch dir | Automatic: linker merges sections |
| Correctness | Stale TMF files possible | Always correct — no statics = no entries |
| Incremental builds | Needs file cleanup logic | Zero-cost: recompile replaces object sections |
| Self-contained | External files alongside PDB | Embedded in binary itself |
| Post-link extraction | Not possible | Any PE parser can read `.trman` section |
| Runtime verification | Not available | Read TRACE_MANIFESTS at runtime from section |
